# Supabase: міграції v3

## ⚠️ Стоп-крапка (дія користувача)

Агент не має доступу до продової БД (тільки anon-ключ), тому міграції
готові, але **застосовуєш їх ти** — у [Supabase Dashboard → SQL Editor](https://supabase.com/dashboard):

1. **Бекап**: Dashboard → Database → Backups → переконатися, що є свіжий.
2. Виконати `migrations/0001_v3_schema.sql` (створює нові таблиці/в'ю; старі не чіпає).
3. Виконати `migrations/0002_backfill_v3.sql`; звірити контрольні суми в
   результаті (рядки `old`/`new` для кожного user_id мають збігтися).
4. Прогнати `tests/rls_check.sql` (перевірка ізоляції).
5. Повідомити агенту/у наступній сесії: «міграції застосовано» — фронт
   перемикається на `work_days_v3` малим PR (список місць — у
   `src/data/README.md`).

Поки міграції не застосовані, застосунок працює через шар `src/data/`
проти старої схеми (один вкладений запит, без N+1) — функціонально
повноцінно; в'ю-агрегати заміщені чистими функціями `src/domain/stats.ts`.

## Каталоги

- `migrations/` — SQL покоління v3 (порядок за номером).
- `tests/` — перевірки RLS.
- `legacy/` — історичні розсипні .sql з кореня репо (застосовані колись
  вручну; збережені для контексту, не запускати).
