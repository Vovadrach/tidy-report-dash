# PRD 4.0 «Ясно» — єдине джерело правди редизайну

> Покоління **4.0**, кодова назва **«Ясно»** (світло · чистота · «усе ясно з грошима»).
> Повна перебудова інфраструктури, архітектури даних і фронтенду.
> Замінює покоління 3.0 «Aria», 2.x «Atelier» і паралельну гілку «Зефір» — **від них не лишається жодної згадки**.
>
> Статус: **чернетка → узгоджено 4 несучі рішення** (див. §3). Готова до фази реалізації.
> Мова інтерфейсу — українська; гроші — євро (€). Платформа — мобільний PWA.

## 1. Візія і принципи

**Що це.** Персональний інструмент самозайнятої прибиральниці: за 15 секунд після зміни записати роботу, з першого погляду бачити скільки зароблено і скільки з того вже сплачено, у один дотик приймати оплату, і мати **живу картку кожного клієнта** — з історією «що і коли я робила, за якими параметрами», якою керуєш звідти ж.

**Чому 4.0.** Це не перефарбування. Ми міняємо одночасно **матеріал** (візуальну мову) і **структуру** екранів/навігації, лікуємо архітектуру даних (прибираємо мертву обгортку `reports`), і виправляємо всі місця, де функції суперечать одна одній. Попередній застосунок зникає повністю: нова назва, нова палітра, нова типографіка, нові класи, нова навігація.

**Три слова відчуття:** *світло · ясно · без зайвого*.

**Принципи проєктування**
- **P1. Один погляд — одна відповідь.** Кожен екран відповідає рівно на одне питання («що я робила», «хто винен», «що з цим клієнтом»). Ніякого дубляжу однієї інформації на трьох екранах.
- **P2. Колір = стан, не декор.** Полотно нейтральне; яскравий колір з'являється лише там, де несе значення — статус оплати і кнопка дії. Без фіолету, без тіней, без підсвіток/glow.
- **P3. Один елемент — одна поведінка.** Години вводяться ОДНИМ віджетом усюди. Оплата виставляється ОДНИМ віджетом усюди. Ніяких трьох різних порядків тих самих кнопок.
- **P4. Мінімум переходів.** Додати запис — один нижній лист, без ланцюжка сторінок. Керувати клієнтом — не покидаючи його картки.
- **P5. Соло за замовчуванням.** Застосунок для однієї людини. Команда/помічниці — прихована опція, що з'являється лише коли справді потрібна.
- **P6. Правильність — у домені.** Уся фінансова математика і інваріанти статусу живуть у чистих доменних функціях і у в'ю Postgres. У компонентах — жодних розрахунків грошей.
- **P7. Швидкість відчутна.** Оптимістичні мутації, миттєвий кеш (offline-read), м'які переходи 150–220 мс. Застосунок ніколи не «висить».

## 2. Користувач, JTBD, скоуп

**Персона (єдина).** Жінка 25–45, професійно прибирає апартаменти в Італії, веде власний облік. Телефон — головний пристрій. Використання ~30–60 с/день, часто стоячи, однією рукою.

**Jobs-To-Be-Done (незмінний кістяк)**
1. Після зміни за ~15 с записати: клієнт + години → сума рахується сама.
2. З першого погляду: скільки зароблено цього місяця і скільки з того сплачено.
3. Побачити хто скільки винен і прийняти оплату (повну/часткову) в один дотик.
4. Відкрити клієнта і побачити **всю історію робіт** з параметрами кожного дня; звідти ж керувати (записати, прийняти оплату, нагадати, змінити ставку).
5. Запланувати майбутню зміну.
6. *(опційно)* Поділити зміну між собою і помічницею й бачити частку кожної.

**У скоупі 4.0:** усе вище; світла тема; PWA; офлайн-читання; повна перебудова UI/навігації; завершення міграції даних v3; жива картка клієнта; уніфіковані елементи годин і оплати.

**Поза скоупом 4.0:** мультивалютність (лише €), мультимова (лише укр. інтерфейс), мультикористувацька синхронізація акаунтів, нативні застосунки. **Черга офлайн-мутацій** — фаза 4.x (у 4.0 офлайн-читання + банер).

## 3. Зафіксовані рішення напряму

Обрано користувачем через візуальні прев'ю (щоб уникнути 5-го відхилення «це те саме»):

| # | Рішення | Обрано | Наслідок для PRD |
|---|---------|--------|------------------|
| D1 | **Центральний екран / скелет** | **Стрічка днів** (хронологічний потік «Сьогодні») | Головна = скрольна стрічка з інлайн-додаванням. Картка клієнта — окремий першокласний екран (вкладка «Клієнти»), не головна. |
| D2 | **Візуальний матеріал** | **Чисті лінії + акценти** | Біле полотно, 1px hairline-межі, велика типографіка. Колір лише на статусах і кнопках дій. Без тіней, без заливок карток, без фіолету. |
| D3 | **Додавання запису / години + гроші** | **Нижній лист, одним кроком** | Глобальний quick-add sheet (vaul). Той самий лист для планування і для «позначити виконаним». |
| D4 | **Команда / помічниці** | **Соло-режим, команда опційно** | За замовчуванням жодного UI помічниць. Поділ роботи/оплати з'являється лише після додавання помічниці у Профілі. |

**Робоча назва покоління:** «Ясно» *(єдине, що не узгоджувалось окремо — легко змінити однією правкою токенів/маніфесту)*.

## 4. Дизайн-мова «Ясно»

### 4.1 ДНК
Дві геометричні першооснови: **лінія** (hairline-межі, тонкі прогрес-лінії) і **крапка** (`StatusDot` — колірна крапка статусу, вона ж мотив бренду). Глибина створюється **контрастом полотна і межами**, а не тінню. Простір — головний матеріал: багато повітря, великий вертикальний ритм.

### 4.2 Палітра (light-only, oklch, БЕЗ фіолету)
Жоден відтінок не лежить у фіолетовому діапазоні (hue 290–330). Бренд — синій (245), статуси — зелений/бурштин/корал/небо.

```css
:root {
  /* Полотно і чорнило */
  --bg:            oklch(0.99  0.003 240);  /* майже біле, ледь прохолодне */
  --surface:       oklch(1     0     0);     /* білі поверхні/картки */
  --surface-inset: oklch(0.975 0.004 240);   /* легкий інсет (поля вводу) */
  --ink:           oklch(0.25  0.02  250);   /* графітове чорнило (синьо-сланець) */
  --ink-2:         oklch(0.52  0.015 250);   /* вторинний текст */
  --ink-3:         oklch(0.68  0.012 250);   /* плейсхолдер/підпис */
  --line:          oklch(0.915 0.005 250);   /* hairline-межа */
  --line-strong:   oklch(0.86  0.006 250);   /* активна межа/розділювач */

  /* Бренд (дія) — яскравий синій, не фіолет */
  --accent:        oklch(0.58  0.17  245);
  --accent-strong: oklch(0.52  0.18  245);   /* натиск/hover */
  --accent-ink:    oklch(0.99  0.01  245);   /* текст на акценті */
  --accent-tint:   oklch(0.95  0.04  245);   /* фон легкого акценту */

  /* Статуси — яскраві, «колоритні, але не кринж» */
  --ok:     oklch(0.63 0.15 155);  /* оплачено — смарагд/м'ята */
  --warn:   oklch(0.74 0.14 78);   /* частково — бурштин */
  --danger: oklch(0.62 0.20 25);   /* не оплачено / борг — корал */
  --info:   oklch(0.66 0.12 230);  /* заплановано — небо */

  /* Радіуси: помірні, «чисті» (не пухкі) */
  --r-card:  0.875rem;  /* 14px */
  --r-field: 0.75rem;   /* 12px */
  --r-sheet: 1.25rem;   /* верхні кути листа */
  --r-pill:  999px;

  /* Рух */
  --ease: cubic-bezier(0.2, 0, 0, 1);
  --dur:  180ms;
}
```

**Правила застосування кольору**
- Гроші **завжди** рендеряться чорнилом `--ink`; символ € — `--ink-3`. Колір суми не залежить від розміру суми.
- Колір несе стан лише через `StatusDot`, статус-пігулку і прогрес-лінію.
- Заповнені кольорові поверхні заборонені (крім: акцентна кнопка дії, FAB, і крихітні tint-пігулки статусу ~4–8% кольору).
- **Жодних** `box-shadow`, `filter: drop-shadow`, `text-shadow`, неонового glow. Floating-елементи (док, лист) відокремлюються hairline-межею і контрастом полотна.

### 4.3 Типографіка
- **Основна:** **Inter** (variable, self-host woff2) — бездоганна кирилиця, tabular-figures, ідеальна під «чисті лінії».
  *Альтернатива для сильнішої ідентичності:* **e-Ukraine** або **Onest** (обидві укр.-дружні, геометричні) — заміна одним `@font-face`-блоком. Див. §15.
- **Числа/суми/години:** той самий Inter із `font-variant-numeric: tabular-nums; letter-spacing: -0.01em`. Великі суми — окремий клас `.display` (щільніший трекінг, вага 600).
- Масштаб (rem): 0.75 / 0.8125 / 0.875 / 1 / 1.25 / 1.75 / 2.5. Заголовки екранів — 1.75; велике число місяця — 2.5.
- Fixel (3.0) і Manrope/Playfair (2.x) **видаляються**.

### 4.4 Іконки
**Lucide** (line-icons, stroke ~1.75) — консистентний штрих під hairline-естетику. `@phosphor-icons/react` **видаляється** (одна система іконок замість двох).

### 4.5 Рух і реакція кнопок («як кожна кнопка реагує»)
| Елемент | Спокій | Натиск (active) | Ефект |
|---------|--------|------------------|-------|
| Кнопка дії (accent) | суцільний `--accent`, білий текст | `scale(0.97)`, `--accent-strong`, 120 мс | тактильний «клац», без тіні |
| Вторинна кнопка | текст `--ink`, межа `--line` | фон `--surface-inset`, `scale(0.98)` | |
| Рядок стрічки / клієнта | білий, hairline-розділювач | фон `--surface-inset` | tap → sheet/route з `viewTransition` |
| FAB «+» | коло `--accent`, білий «+» | `scale(0.92)`, легкий пружний відскок | відкриває quick-add лист (slide-up spring) |
| StatusDot | кільце/диск статусу | pop-scale при зміні статусу | миттєвий фідбек оплати |
| Свайп рядка | — | вправо=оплатити (зелена підкладка), вліво=видалити (коралова) | поріг 96px, друг. фідбек іконкою |
| Нижній лист | — | drag-to-dismiss (vaul), пружина | затемнення полотна 8–12% (не чорне) |
| Перехід маршруту | — | — | View Transitions API (Router 7 `viewTransition`), крос-фейд 180 мс |
| Число, що змінилось | — | — | короткий count-up tween (необов'язково, тактовно) |

- Тривалість базова `--dur: 180ms`, easing `--ease`. Довше 240 мс — заборонено (відчуття «гальмує»).
- Поважати `prefers-reduced-motion`: вимикати count-up і великі translate, лишати миттєві кросфейди.

## 5. Інформаційна архітектура і навігація

### 5.1 Карта
```
Auth ─ /login, /register
Профіль ─ /profile (аватар у шапці)
│
├─ Стрічка        /            (головна: потік днів)     ─┐
├─ Клієнти        /clients      → Картка клієнта /client/:id │ нижній
├─ Гроші          /money        (Борги | Огляд)              │ док
│
└─ FAB ( + )      → QuickAddSheet (глобальний лист)  ────────┘
```
Глибокі/модальні: `DaySheet` (деталь дня), `PaymentSheet` (прийняти оплату), `QuickAddSheet` (додати/запланувати/виконати). Усі — нижні листи (vaul), не окремі повноекранні сторінки.

### 5.2 Нижній док (єдиний компонент)
- Три рівновіддалені призначення: **Стрічка** · **Клієнти** · **Гроші** (текст + Lucide-іконка, активний = `--accent`, решта = `--ink-3`).
- **FAB «+»** — окреме кругле коло `--accent`, підняте над доком, горизонтально по центру; відкриває `QuickAddSheet`.
- Обгортка дока `pointer-events: none`, клікабельні лише елементи (успадкований урок: смуга не має перехоплювати тапи по контенту).
- Без «чорнильного» дока Aria і без градієнтної FAB Зефіра — тут світлий hairline-док і плаский синій FAB.

### 5.3 Правила переходів
- `Row` (клієнт/день) → відповідний sheet (не нова сторінка), крім вкладки «Клієнти» → повноекранна **Картка клієнта** (`/client/:id`, `viewTransition`).
- Ім'я клієнта будь-де (стрічка, борги) — глибоке посилання на його картку.
- Немає «висячих» параметрів (`returnTo` без обробника) і немає екранів без виходу.

## 6. Поекранна специфікація

Кожен екран описано форматом: **Мета · Макет · Елементи · Стани · Взаємодії**. Жоден екран не лишається без нової архітектури.

### 6.1 Стрічка (`/`) — головна
- **Мета:** що я робила і що з оплатою; миттєво додати запис. (JTBD 1–2.)
- **Макет:** липка `app-bar` (назва місяця, велика сума заробленого, `LineProgress` сплачено; аватар→Профіль). Тіло — вертикальна стрічка, згрупована за днями (реверс-хронологія), «Сьогодні» завжди зверху. Групи роздільник — hairline + `day-heading`.
- **Елементи:** `Row` на кожен запис (`StatusDot` · імʼя клієнта · `HoursBadge` · сума). `SwipeableRow` (вправо=оплатити повністю, вліво=видалити з `ConfirmSheet`; для планових вимкнено). Перемикач періоду в шапці (місяць ‹ ›, тап на назву → вибір місяця/року/«весь рік»). Pull-to-refresh (`usePullToRefresh`).
- **Стани:** skeleton при завантаженні; inline-retry при помилці; `EmptyState` «Ще немає записів — додай перший» з CTA→FAB (без суперечності «порожньо, але сьогодні є»: якщо записів 0 — показуємо чистий EmptyState, а не самотню сьогоднішню бульбашку).
- **Взаємодії:** тап рядка → `DaySheet`. Плановий рядок (пунктирний, `--info`) → `QuickAddSheet` у режимі «Позначити виконаним». Тап дати-заголовка «+» → `QuickAddSheet` з предзаповненою датою.

### 6.2 QuickAddSheet (глобальний нижній лист) — D3
- **Мета:** за один крок додати/запланувати/виконати запис; тут живуть **обидва** перевикористовувані елементи — години і гроші.
- **Макет (vaul, slide-up):** заголовок «Новий запис / Позначити виконаним» + ✕; поле **Клієнт** (комбобокс з пошуком; або предзаповнено); **`HoursField`** (степер −/+ 15 хв, тап→компактний ввід); **`AmountField`** (рахується як години×ставка; тап-перемикач «від годин ↔ від суми»); **`PaymentControl`** (статус: Оплачено / Частково(+сума) / Не оплачено); опційно нотатка; опційно (лише якщо є помічниці) кнопка «Поділити». Дата — компактний контрол (сьогодні за замовч., тап→нативний пікер).
- **Первинна дія:** «Зберегти». Якщо години = 0 і дата в майбутньому → кнопка стає «Запланувати». Якщо відкрито з планового запису → «Позначити виконаним» (проставляє години, `isPlanned=false` однією мутацією).
- **Стани:** валідація інлайн (сума не може перевищувати, години обовʼязкові для «Зберегти»); кнопка недоступна доки не валідно; оптимістичний `onSuccess` закриває лист.
- **Взаємодії:** відкривається з FAB, з дати-заголовка стрічки, з картки клієнта («Записати роботу»), з планового рядка.

### 6.3 Клієнти (`/clients`)
- **Мета:** список клієнтів як точок входу в картки; керування списком.
- **Макет:** пошук; список `Row` (імʼя · «€/год» · компактний баланс місяця + `StatusDot`); кнопка «Додати клієнта» (лист).
- **Стани:** skeleton; empty «Немає клієнтів» з CTA; onboarding-варіант (перший запуск).
- **Взаємодії:** тап → **Картка клієнта**. Свайп/меню рядка → Редагувати / Архівувати.

### 6.4 Картка клієнта (`/client/:id`) — ключова нова фіча
- **Мета:** «що і коли я робила для цього клієнта, за якими параметрами — і звідти керувати». (JTBD 4.)
- **Макет (повний екран, `viewTransition`):**
  1. **Шапка:** імʼя; ставка (інлайн-редагована); підсумок балансу — зароблено (всього / цей місяць), сплачено, **борг** зі `StatusDot` і `LineProgress`.
  2. **Панель дій:** `Записати роботу` (→ QuickAddSheet предзаповнений) · `Прийняти оплату` (→ PaymentSheet) · `Нагадати` (navigator.share/копіювання) · `Редагувати` · `Архівувати`.
  3. **Історія (таймлайн):** ВСІ робочі дні клієнта, реверс-хронологія; кожен рядок — дата, `HoursBadge`, застосована ставка, сума, `StatusDot`, помічниці (якщо є), нотатка (згорнута). Фільтр-сегменти: **Усі · Борг · Цей місяць · Рік ▾**.

### 6.5 Гроші (`/money`) — Борги + Огляд (єдиний екран, сегменти)
- **Мета:** хто винен + приймання оплат (JTBD 3) і аналітика/дашборд — **без дубляжу** боргів по інших екранах.
- **Сегмент «Борги»:** велике «Тобі винні X €» + несплачені години; список боржників (спадання боргу), кожен → `PaymentSheet`. Кнопка «Нагадати» на рядку.
- **Сегмент «Огляд» (дашборд):** плитки місяця/періоду (Зароблено · Години · Сплачено · Залишок) з `PeriodChips` (Місяць/Рік/Весь час); `EarningsChart` (6 міс., власний SVG, faint=зароблено, суцільний=сплачено); «Топ клієнтів». **Борги показуються лише тут і в картці клієнта** — на стрічці і в огляді немає окремих клікабельних/неклікабельних дублів.
- **Стани:** empty «Все сплачено 🎉»; skeleton; inline-retry.

### 6.6 PaymentSheet (нижній лист) — канонічне «застосування грошей»
- **Мета:** прийняти оплату одним контролом усюди однаково.
- **Макет:** сума боргу/дня зверху; чипи швидкого вводу **Уся сума · +50 · +100 · власна**; живий «Залишок»; підтвердити. Розподіл оплати між днями (найстаріші першими) — доменна функція `distributePayment` (нова, див. §8), якщо оплата стосується кількох днів клієнта.
- **Правила:** сума > боргу відхиляється з поясненням; статус перераховується через `resolveStatus`; оптимістично.

### 6.7 DaySheet (нижній лист) — деталь/редагування дня
- **Мета:** переглянути й змінити один день без окремої сторінки й без бага автозбереження.
- **Макет:** клієнт · дата · **`HoursField`** · **`AmountField`** · **`PaymentControl`** · помічниці (якщо є) · нотатка · Видалити.
- **Збереження:** **явна** мутація «Зберегти» (оптимістична), НЕ автозбереження-на-unmount. Зміна годин ЗАВЖДИ реконсилює `paidAmount`+`status` (див. §9, баг B1).
- **Стани:** «Дані не знайдено» якщо день зник; skeleton.

### 6.8 Профіль/Налаштування (`/profile`)
- **Мета:** акаунт і рідко-змінні налаштування (виправляє «вихід захований у виборі клієнта»).
- **Елементи:** email/акаунт; **Помічниці** (додати/видалити — саме тут вмикається команда, D4); ставка за замовчуванням; експорт даних (CSV/JSON); тема (світла — єдина у 4.0); версія/збірка; **Вийти**.
- Доступ: аватар у шапці стрічки.

### 6.9 Auth (`/login`, `/register`)
Рестайл під «Ясно»: чисте полотно, один акцентний CTA, крос-лінки, валідація (пароль ≥6, збіг). Після реєстрації → `/login`; після входу → `/`.

### 6.10 Системні стани
- **Onboarding (перший запуск, 0 клієнтів):** привітання + «Додай першого клієнта» → лист. Після — підказка «Тепер натисни +, щоб записати роботу».
- **Empty / Loading / Error / Offline:** уніфіковані `EmptyState`, `Skeleton`, inline-retry, `OfflineBanner` (тонка лінія зверху, не блокує).
- **404 (`*`):** чиста картка + кнопка «На головну» (SPA-навігація, без повного релоуду).

## 8. Доменна модель і бізнес-правила

Канонічна модель — camelCase (без snake_case поза шаром `data/mappers`). Після міграції v3 з `WorkDay` **зникає** перехідне поле `reportId`.

```ts
type PaymentStatus = "paid" | "partial" | "unpaid";
type ISODate = string; // 'YYYY-MM-DD', ЛОКАЛЬНА дата

interface Client   { id; name; hourlyRate: number; archivedAt?: string | null }
interface WorkDay  { id; clientId; date: ISODate; hours: number; amount: number;
                     paidAmount: number; status: PaymentStatus; isPlanned: boolean;
                     note?: string; assignments: Assignment[] }
interface Assignment { id; workerId: string|null; workerName; workerColor; hours; amount }
interface Worker   { id; name; color; isPrimary: boolean }
```

**Правила**
- **R1. Сума:** `amount = round2(hours × client.hourlyRate)`; або користувач вводить суму вручну і години back-solve. Уся математика — `domain/money.ts` або в'ю Postgres; у компонентах — нуль.
- **R2. Інваріант статусу:** `status = resolveStatus(paidAmount, amount)` — `0→unpaid`, `<amount→partial`, `≥amount→paid`. Один шлях запису; **не** тригер БД. `paidAmount` завжди `min(paidAmount, amount)`.
- **R3. Години:** зберігаються `numeric(6,2)`; ввід кроком 15 хв; конверсія «Г:ХВ»⇄десяткові лише в `domain/time.ts`.
- **R4. Планові дні:** `isPlanned=true` — виключені з усіх агрегатів (`filter (not is_planned)`). Одна кнопка перемикає намір; «Позначити виконаним» = проставити години + `isPlanned=false` однією мутацією.
- **R5. Борг:** `totalDue = Σ(amount − min(paidAmount, amount))` по неоплачених непланових днях; `unpaidHours` — пропорційно неоплаченій частці кожного дня (в'ю `client_balances`).
- **R6. Розподіл оплати (нове, `distributePayment`):** лумп-сума клієнта кладеться на **найстаріші несплачені дні першими**, поки не вичерпається; залишок — на частковий останній день. Покривається vitest-таблицею.
- **R7. Помічниці (опційно, D4):** за замовчуванням `assignments=[]` і UI помічниць прихований. Якщо додано ≥1 помічницю — доступний поділ: `Σ часток = amount` (інакше підтвердження недоступне); валідація і поділ рахуються від **фінальної** суми (див. §9, баг B2). Перша додана помічниця стає `isPrimary`.
- **R8. Нагадування:** `reminderMessage(balance)` → шаблон «Добрий день! За прибирання: N год, борг X €. Дякую!»; `navigator.share`, fallback — копіювання + toast.
- **R9. Дати:** усі локальні; `toISOString().split` заборонено поза `domain/dates.ts` (lint-правило) — фікс UTC-«сьогодні».

## 10. Архітектура даних і міграція

### 10.1 Ціль: завершити модель v3 (ядро «перебудови архітектури»)
Прибрати мертву обгортку `reports` (1 report = 1 день — рудимент). Одиниця обліку — **робочий день**.

**Нова схема (вже написана у `supabase/migrations/`):**
- `work_days_v3(id, user_id, client_id, date, hours, amount, payment_status, paid_amount, is_planned, note, created_at)` — `user_id` і `client_id` **прямо** на дні (тривіальна RLS, без дублю `client_name`).
- `work_day_assignments_v3(id, work_day_id, worker_id, deleted_worker_name, hours, amount)`.
- `clients.archived_at` — мʼяке архівування.
- В'ю (`security_invoker=true`): **`client_balances`** (total_earned/hours/paid/due, unpaid_hours пропорційно) і **`monthly_summary`** (hours/earned/paid по місяцях). **Нове в'ю `client_history`** (усі дні клієнта з полями для таймлайну картки) — додати у міграцію 0003.
- Інваріант статусу — доменом (`resolveStatus`), не тригером.

### 10.2 Runbook міграції (дія користувача — потрібен доступ до прод-БД)
1. **Бекап** БД (Supabase → Database → Backups / `pg_dump`).
2. Виконати `supabase/migrations/0001_v3_schema.sql` (створює v3-таблиці, RLS, в'ю).
3. Виконати `0002_backfill_v3.sql` (переносить дані; ідемпотентно; звірити контрольні суми old vs new per user — мають збігтися `count`, `Σamount`, `Σpaid`).
4. Виконати новий `0003_client_history_view.sql`.
5. Прогнати `supabase/tests/rls_check.sql` (anon читає 0 рядків / insert падає; чужий user — 0 рядків; в'ю успадковують RLS).
6. Повідомити: «міграції застосовано» → вмикаємо фронт-перемикання.
7. Старі таблиці (`reports`, `work_days`, `work_day_assignments`) лишити read-only архівом до окремого підтвердження на DROP.

### 10.3 Шар даних фронту
- **`Backend` інтерфейс** (абстракція) лишається: `backend.supabase.ts` (прод) + `demo/fixture.ts` (`VITE_DEMO=1`).
- Після міграції `backend.supabase.ts` переходить на `work_days_v3` одним вкладеним запитом: `from("work_days_v3").select("*, work_day_assignments_v3(*, worker:workers(*))")`; `createWorkEntry` — один insert (без report); `deleteReport`/`reportId` видаляються.
- Баланси/історія — з в'ю `client_balances`/`monthly_summary`/`client_history` (контракти = чинним `domain/stats.ts`, тож перемикання — малий PR).
- **Типи Supabase — генеровані** (`supabase gen types typescript`), не рукописні (фікс дрейфу схеми).
- Валідація меж БД→домен — `zod` у `mappers.ts` (єдине місце snake_case).

## 12. Нефункціональні вимоги

- **Продуктивність:** перший чанк ≤ 150 KB gzip; головна = **1** мережевий запит; TTI < 2 с на 4G/середньому Android; роут-спліт (lazy) усіх екранів крім головної.

- **Локаль:** укр. інтерфейс, €-суми, локальні дати (Europe/Rome як TZ обліку), tabular-цифри.
- **Безпека:** RLS `auth.uid() = user_id` прямо на `work_days_v3` (тривіально), assignments — через власника дня; anon не читає нічого; ключі лише в env.
- **Приватність/експорт:** експорт власних даних (CSV/JSON) з Профілю.

## 15. Відкриті питання

- **🔎-1 Назва покоління.** «Ясно» — робоча. Альтернативи: «Промінь», «Крапка» (мотив status-dot), «Свіжо». Рішення користувача (легко змінити).
